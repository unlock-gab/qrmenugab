## Main Phase 6 modules

### 1. Light inventory / stock foundation

Add a practical and lightweight stock management layer.

Goal:
Restaurant owners should be able to control whether an item is:

* available
* unavailable
* optionally stock-tracked with a quantity

This is not a full warehouse ERP.
Keep it simple and operational.

Suggested support:

* item availability toggle
* optional stock quantity
* optional stock tracking enabled/disabled per item
* automatic item disable when stock reaches zero if stock tracking is enabled

Requirements:

* menu items can still be simple if stock tracking is not enabled
* if tracking is enabled, order creation must respect stock quantity
* unavailable or out-of-stock items should be clearly shown on the public menu
* merchant dashboard should allow easy stock updates

Do not overcomplicate with supplier/purchase order logic.

---

### 2. Ingredients and item options structure

Add a cleaner product composition layer.

Goal:
Allow menu items to support:

* ingredients display
* optional modifiers / add-ons / choices
* simple customization for customers

Examples:

* extra cheese
* no sugar
* spicy level
* size options
* drink add-ons
* flavor choice

Requirements:

* merchant can manage options/modifiers for a menu item
* public menu can display selectable options where relevant
* options can affect price where needed
* order details must capture selected options clearly
* kitchen/waiter/cashier views should display option choices in a readable way

Optional ingredients display:

* show ingredients text/list on product page/card
* support allergen-friendly or composition notes if useful

Do not build a huge POS modifier engine.
Keep it simple but structured correctly.

---

### 3. Promo code and discount system

Add a practical first version of discounts and promo codes.

Goal:
Allow restaurants to run simple promotions and discounts.

Suggested features:

* promo code creation in merchant dashboard
* fixed discount or percentage discount
* active/inactive toggle
* start/end dates
* optional usage limit
* optional minimum order amount

Public ordering flow:

* customer can enter promo code
* code gets validated
* cart/order total updates correctly
* invalid code handling is clear
* used discount is stored with the order

Order details and receipt:

* show discount amount
* show final total after discount

Keep the system clean and safe.
Do not implement overly complex coupon stacking logic.

---

### 4. Call waiter feature

Add a customer-facing **Call Waiter** function.

Goal:
A customer sitting at a table can request attention without placing a new order.

Examples:

* call waiter
* request bill
* ask for assistance
* ask for water or service attention

Requirements:

* public customer menu/table context should allow a simple request action
* request should be linked to restaurant + table
* waiter/merchant operational interface should receive the request
* request should be visible and actionable
* requests should be markable as handled/resolved

Possible request types:

* CALL_WAITER
* REQUEST_BILL
* HELP

Keep it practical.
Do not make it too complex.
This feature can be very valuable for real cafés.

### 5. Reservation foundation

Suggested support:

  * time
  * guest count
  * optional notes

Suggested statuses:

* PENDING
* CONFIRMED
* CANCELLED
* COMPLETED

Requirements:

* reservation management page in merchant dashboard
* basic calendar/list view or filtered list
* clean validation
* conflict handling can be simple in this phase
* structure it cleanly for future expansion

Do not build a massive scheduling system yet.
Just create a clean and useful reservation base.

## Business logic and validation rules

### Stock

* if stock tracking disabled, item behaves normally
* if stock tracking enabled, stock must be respected at order time
* stock should decrement safely when order is placed if that is the chosen logic
* handle race conditions reasonably for MVP/SaaS scale

### Promo codes

* validate date range
* validate active status
* validate usage limit
* validate minimum order amount
* prevent invalid discount totals

### Waiter requests

* require valid table context
* scope to restaurant/table
* operational staff can mark resolved

### Reservations

* validate date/time
* validate guest count
* handle basic merchant confirmation flow cleanly

### Multilingual

* fallback safely when translation missing
* avoid breaking old menu items with no translation data

Keep logic centralized and maintainable.

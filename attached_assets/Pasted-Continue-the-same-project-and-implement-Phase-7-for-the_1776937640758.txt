## Main Phase 7 modules

### 1. Multi-branch restaurant architecture

Upgrade the data model so a restaurant brand can have one or more **branches/locations**.

Goal:
A single merchant organization can manage multiple physical locations, each with its own:

* tables
* orders
* kitchen activity
* waiter/cashier activity
* settings
* staff
* QR routes
* reservations
* stock/menu availability if needed

Important design direction:

* keep the top-level restaurant/organization concept
* introduce a branch/location layer below it
* scope operational records to the branch where appropriate
* do not break existing single-location restaurants

Suggested architecture:

* Restaurant = organization/brand/account
* Branch = physical location

Single-branch restaurants should still work cleanly with minimal friction.

---

### 2. Branch-level scoping

Refine the platform so branch-sensitive data is scoped properly.

Records that likely need branch relation:

* tables
* orders
* staff assignments
* reservations
* waiter requests
* kitchen/waiter/cashier views
* stock quantities if stock is branch-specific
* possibly menu availability if certain items differ by branch

Requirements:

* merchant users should only see allowed branch data
* owner can see all branches under their restaurant organization
* branch staff sees only their assigned branch
* public dine-in QR flow resolves correctly to the right branch + table
* no cross-branch leakage

This scoping must be clean and safe.

---

### 3. Branch management UI

Add branch management tools for merchant owners.

Merchant owner should be able to:

* create branch
* edit branch
* deactivate branch if needed
* set branch name
* set branch address
* set phone
* set operating info
* set branch status
* optionally configure branch-specific branding/settings if useful later

Create a clean branch management page inside merchant dashboard.

Also add a clear branch switcher or branch selector UX for owner users where helpful.

---

### 4. Delivery and takeaway order modes

Until now, the system focused mainly on dine-in QR table ordering.
Now extend it to support:

* **DINE_IN**
* **TAKEAWAY**
* **DELIVERY**

Goal:
A restaurant can receive not only table QR orders but also direct non-table customer orders.

Requirements:

* public ordering flow should support branch-specific takeaway/delivery ordering
* dine-in flow must keep table context
* takeaway/delivery flow must work without table context
* order type must be stored clearly
* operational interfaces should display order type clearly

Delivery orders may include:

* customer name
* phone
* address
* optional note

Takeaway orders may include:

* customer name
* phone
* optional note

Do not build a full delivery logistics platform.
Keep it focused on order intake and restaurant handling.

---

### 5. Public ordering mode selection

Add a clean public ordering entry experience.

For a branch, users may be able to:

* scan table QR and order dine-in directly
* open a public branch order page and choose takeaway
* open a public branch order page and choose delivery

Requirements:

* ordering UX should stay simple and mobile-friendly
* branch identity should remain clear
* order mode should be obvious throughout checkout
* totals and order data should remain accurate

Do not make the public flow confusing.

### 6. Customer accounts foundation

Add optional customer account support.

Goal:

This should be optional and lightweight.

Suggested capabilities:

* customer can log in and view past orders
* future loyalty can attach to customer account

Keep the architecture clean and safe.

## Business logic and validation rules

### Multi-branch

* owner can access all branches of their restaurant
* branch staff only their assigned branch
* branch deactivation handled gracefully
* existing single-branch restaurants remain functional

### Order types

* DINE_IN requires table context
* TAKEAWAY does not require table
* DELIVERY requires customer + address fields
* operational views must handle each mode correctly

### Customer accounts

* guest ordering can still exist
* authenticated customers can view their own history only
* customer data must not leak across restaurants inappropriately

### Loyalty

* only eligible paid orders should add points
* points logic should be centralized
* keep reward logic simple and extensible

### Notifications

* notification events should be structured
* actual provider send logic should be abstracted
* failures should be logged cleanly if implemented

Keep logic centralized and maintainable.

Build a production-minded **Phase 1 architecture** for a multi-restaurant QR ordering web app that is being developed inside Replit first, but must be structured from day one to be easily migrated later to a **VPS using Dokploy and Docker**.

## Core product idea

This platform is for cafés and restaurants. Each table has its own QR code. When a customer scans the QR code, they are taken directly to that restaurant’s digital menu for that specific table. The customer can browse menu items, add items to cart, and place an order. The restaurant owner or staff should receive the order inside their dashboard with a clear sound notification, full order details, and the table number.

This Phase 1 is only about building the correct **architecture foundation + MVP base**, not the full final SaaS yet.

---

Create a solid, scalable, Docker-friendly foundation with:

* clean architecture
* multi-restaurant support
* merchant dashboard base
* customer ordering flow base

## Architecture requirements

Design the project as a **multi-tenant restaurant ordering platform** where each restaurant has its own:

* account
* tables
* categories
* menu items
* orders

The architecture must be simple, scalable, and easy to extend in later phases.

## Phase 1 scope only

Implement only these modules now:

### 1. Authentication base

* merchant login page
* secure authentication structure
* role-ready architecture even if only one merchant role is used now
* prepare for future admin and staff roles
* do not overbuild permissions yet, just structure it cleanly

### 2. Restaurant entity

Each merchant account should belong to one restaurant.
Restaurant fields should include:

* name
* slug
* phone
* address
* logo
* status
* createdAt
* updatedAt

### 3. Tables management

Each restaurant can manage its tables.
Fields:

* table number
* QR token
* active/inactive
* relation to restaurant

Create a clean merchant dashboard page to:

* list tables
* create a table
* edit table
* activate/deactivate table
* view QR value/token
* prepare structure for future QR image generation

### 4. Categories management

Each restaurant can manage menu categories.
Fields:

* name
* sort order
* active/inactive
* restaurant relation

Create dashboard CRUD for categories.

### 5. Menu items management

Each restaurant can manage menu items.
Fields:

* category
* name
* description
* price
* image URL
* availability status
* sort order
* restaurant relation

Create dashboard CRUD for menu items.

### 6. Customer QR route

When a customer scans a QR code, the URL should open a public customer page tied to:

* restaurant
* table

Design clean routing logic such as:

* restaurant slug
* table token
  or an equivalent scalable public route structure

The public page must:

* identify restaurant
* identify table
* load the correct menu
* keep table context during the session

### 7. Customer menu page

Build a clean mobile-first customer menu experience:

* restaurant branding
* category sections
* menu cards
* item image
* item name
* price
* availability state
* add to cart button
* quantity controls
* cart summary
* total price

This should feel modern, premium, and optimized for phone use.

### 8. Cart and order creation

Allow the customer to:

* add items
* remove items
* change quantities
* add optional order note
* see total
* submit order

When the customer submits:

* create an order linked to restaurant and table
* create order items
* assign initial status such as `NEW`
* save totals correctly

No payment integration yet.

### 9. Merchant orders page

Build the first merchant orders dashboard page.
Merchant must see:

* new orders
* order number
* table number
* item list
* quantities
* notes
* total
* order time
* status

Allow merchant to update order status with a simple flow like:

* NEW
* PREPARING
* READY
* SERVED
* PAID
* CANCELLED

For now, if real-time sockets are too heavy for Phase 1, use a clean implementation that is easy to upgrade later.
But structure the code so real-time updates and live notifications can be added in Phase 2 without refactoring everything.

### 10. Notification-ready architecture

Do not fully build advanced live notifications yet unless lightweight and stable.
But structure the merchant orders page and backend so that:

* real-time order updates
* sound notifications
* push-like behavior
  can be plugged in easily in Phase 2.

## Multi-tenant safety

The app must be built so that one restaurant never sees another restaurant’s data.
Every dashboard query must be scoped to the authenticated merchant’s restaurant.
Every public menu and order must be scoped correctly to the restaurant and table in the QR route.

This is critical.

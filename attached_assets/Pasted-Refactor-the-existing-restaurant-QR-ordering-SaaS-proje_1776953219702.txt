Refactor the existing restaurant QR ordering SaaS project into a **clear 3-space architecture** with:

1. **Public Homepage / Visitor Space**
2. **Merchant Space (restaurants and cafés management)**
3. **Platform Admin Space**

Important: this project started in Replit, but it is intentionally being built so it can later be migrated to a **VPS using Dokploy and Docker**.
Keep everything **portable, production-minded, and Docker-friendly**.
Do not introduce Replit-only architecture.
Do not break the existing foundations already built for:

* QR ordering
* menu management
* merchant dashboard logic
* admin logic
* subscriptions
* branch support
* French-first language structure
* Arabic secondary language support
* DA currency formatting

This refactor is mainly about **reorganizing the product experience and route structure** so the platform feels like a real SaaS + public marketplace product instead of opening directly on a merchant login page.

---

## Mandatory language and currency rules

These rules are required across the whole project:

* **French is the default language everywhere**
* **Arabic is the secondary switchable language**
* default locale must be **French**
* if Arabic content is missing, fallback to **French**
* the whole UI must be stable in both French and Arabic
* all visible prices must use **DA**
* currency display examples:

  * `250 DA`
  * `1 200 DA`
  * `3 500 DA`

This applies to:

* public homepage
* restaurant public pages
* customer ordering flow
* merchant dashboard
* admin space
* subscription pages
* reports and billing

## Main objective of this refactor

The app must no longer feel like it starts with a login page.

Instead, the platform should now have:

### A. Public visitor/customer-facing space

This is the default entry point for anyone visiting the website.

Users should be able to:

* discover registered restaurants and cafés
* browse featured establishments
* view trending items or trending restaurants
* search and filter by city/category
* open public restaurant pages
* access public ordering flows
* understand the platform product
* discover pricing / features if relevant

### B. Merchant space

This is for restaurant/café owners and staff to manage their business.

They should access:

* merchant login
* merchant onboarding
* merchant dashboard
* menu management
* tables
* orders
* staff
* settings
* subscription

### C. Platform admin space

This is the internal control center for the platform owner.

Admin should access:

* admin login
* admin dashboard
* restaurants management
* subscriptions
* plans
* users
* leads / demo requests
* platform-wide analytics and control

## Refactor goal

Keep all existing business logic and core modules where possible, but **restructure routes, layout hierarchy, navigation, entry flow, and UI organization** so the product feels like a real public-facing SaaS platform with separate merchant and admin workspaces.

Do not rebuild from scratch.
Preserve existing working modules unless refactoring is necessary to support the new structure cleanly.

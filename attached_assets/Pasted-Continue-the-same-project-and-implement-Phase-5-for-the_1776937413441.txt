## Main Phase 5 modules

### 1. Kitchen Display System (KDS)

Add a dedicated **kitchen display mode** for restaurant staff.

Goal:
Kitchen staff should be able to see incoming food/drink orders in a focused operational interface optimized for speed and clarity.

Requirements:

* dedicated kitchen route/page or workspace
* show relevant active orders only
* strong visibility of:

  * order number
  * table number
  * order age / time since placed
  * items
  * item quantities
  * notes
  * current status
* kitchen-friendly large readable layout
* easy status actions such as:

  * NEW
  * PREPARING
  * READY

The KDS should be visually optimized for continuous operational use on a screen or tablet.

Do not clutter it with subscription/billing/admin concerns.
Make it operational and fast.

---

### 2. KDS filtering and routing

Examples:

* show only active food prep orders
* hide completed/paid/cancelled orders
* optional filtering by status
* optional grouping by newest/oldest
* clear separation of urgent vs older orders

If useful, add:

* full-screen mode friendly layout
* compact vs expanded order cards
* highlight for newly arrived orders

Do not overcomplicate with too many controls.
Keep it practical for real service use.

### 3. Waiter mode

Add a dedicated **waiter mode** or waiter workspace.

Goal:
Waiters should be able to handle front-of-house activity in a clean operational interface.

Possible responsibilities:

* view live table orders
* check order status per table
* mark orders as served
* view active tables
* see if a table has unpaid orders
* possibly create manual internal orders if allowed

Minimum requirements:

* waiter sees restaurant-scoped orders/tables only
* easy table-centric interface
* quick access to:

  * table number
  * active orders
  * order status
  * total amount
  * payment state if relevant

The waiter UI should be optimized for quick use on tablet/mobile or a compact desktop view.

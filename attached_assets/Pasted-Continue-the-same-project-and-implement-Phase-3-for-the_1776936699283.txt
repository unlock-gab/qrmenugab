## Main Phase 3 modules

### 1. Platform admin panel

Create a true platform-level admin area separated from the merchant dashboard.

Admin panel must allow the platform owner to:

* view all restaurants
* create restaurant accounts
* edit restaurant basic information
* activate/deactivate restaurants
* suspend restaurants if needed
* view subscription status
* view merchant account info
* access platform metrics overview

Important:

* keep platform admin clearly separated from merchant space
* do not allow merchants to access admin routes
* admin routes must be protected by role
* admin UI should feel premium, clean, and operational

Suggested admin sections:

* dashboard
* restaurants
* merchants/users
* subscriptions
* plans
* platform settings

---

### 2. Better role system foundation

Upgrade the auth and role structure to support:

* PLATFORM_ADMIN
* MERCHANT_OWNER
* MERCHANT_STAFF

You do not need to implement a fully complex permission matrix yet.
But structure the app cleanly so route protection and future role expansion are easy.

Requirements:

* platform admin can manage all restaurants
* merchant owner manages only their own restaurant
* merchant staff is scoped to one restaurant and has limited operational access
* route guards and server-side checks must be properly scoped

Do not scatter permission logic randomly in components.

---

### 3. Merchant onboarding flow

Add a clean merchant onboarding flow for new restaurant accounts.

Goal:
a new restaurant/business can be created with the essential setup needed to start using the system.

Suggested onboarding steps:

* restaurant basic info
* create owner account
* create first tables
* create first categories
* create first menu items
* ready state for launch

This onboarding does not need to be extremely long or fancy.
But it should feel guided, clear, and useful.

Possible implementations:

* platform admin creates merchant and restaurant, then merchant completes onboarding
  or
* merchant self-signup exists in a controlled form and awaits activation

Choose the cleaner structure that fits the existing project best.

---

### 4. Restaurant lifecycle and status management

Expand restaurant status logic so the platform admin can control account state.

Suggested statuses:

* ACTIVE
* INACTIVE
* SUSPENDED
* PENDING_SETUP

Requirements:

* admin can change restaurant status
* inactive or suspended restaurants should not operate normally
* public QR ordering route must respect restaurant status
* merchant dashboard access must respect restaurant status
* handle blocked states gracefully with proper UI messaging

Do not create confusing behavior.
Make status handling consistent across the platform.

---
